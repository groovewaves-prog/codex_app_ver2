# 業界標準・公的ガイドライン定義の外部化方針

## 目的

現状のレビュー基準は `secure_review/rubric.py` に Python 定数として実装されている。これにより動作は安定している一方、業界標準・公的ガイドラインを追加・更新するたびにコード改修が必要になる。

今後は、レビュー基準をコードから分離し、標準名・バージョン・対象文書種別・章構造・チェック項目・重要度・根拠をデータとして管理できる形にする。

## 基本方針

1. レビュー基準は `standards/` 配下の YAML または JSON に外部化する。
2. 各基準には、標準名、正式バージョン、参照元、適用対象、最終確認日を必ず持たせる。
3. LLM プロンプトには全基準を無条件投入せず、文書種別とレビュー観点に合う基準だけを選択して投入する。
4. 欠落章や必須要素不足の一次判定は、LLM の自由記述ではなく構造化データに基づく決定的ロジックで行う。
5. LLM は、一次判定の説明補助、影響整理、推奨対応の具体化に使う。

## 想定フォーマット

```yaml
standard_id: ipa_design_agreement
name: IPA 機能要件の合意形成ガイド
version: "review-tool-profile-2026-05"
source_url: ""
last_reviewed: "2026-05-13"
applies_to:
  - design
chapters:
  - chapter_id: ch1
    chapter_name: はじめに
    purpose: 目的・スコープ・関係者の明示
    required: true
    items:
      - item_id: "1.1"
        name: 本書の目的
        necessity: must
        expected_content: 構築目的、対象システム、想定読者、期待される成果
        evidence_keywords:
          - 目的
          - 対象システム
          - 想定読者
```

## 更新手順

1. 追加・更新する標準の公式文書とバージョンを確認する。
2. `standards/` 配下に標準定義ファイルを追加または更新する。
3. 既存のテスト文書で、過剰指摘と見落としがないか確認する。
4. `tests/` に、最低限以下のテストを追加する。
5. 欠落章が正しく検出されること。
6. 対象外の章が重要欠落扱いされないこと。
7. 標準のバージョン変更で既存レビュー結果が大きく崩れないこと。

## 追加候補の考え方

新しい標準は、増やせば良いものではない。基準が増えすぎるとプロンプトが肥大化し、LLM のトークン上限・応答揺れ・過剰指摘につながる。

追加する場合は、次のいずれかを満たすものを優先する。

- 文書種別に直接対応する。
- 既存基準では見落としやすいリスクを補える。
- チェック項目を構造化しやすい。
- 公式にバージョン管理され、更新確認が容易である。

## 当面の運用

当面は現在の `rubric.py` 実装を正とし、外部化は段階的に進める。まずは設計書構造定義を外部化し、その後に手順書・運用文書・提案書へ展開する。
