# 技術文書レビュー支援ツール モック操作マニュアル

Last updated: 2026-05-29

この文書は、上司・評価者に Streamlit モックを試用してもらうための操作マニュアルです。
開発者向けの内部仕様ではなく、画面を見ながら一巡できることを目的にしています。

G-4.5 以降、アプリ名 `技術文書レビュー支援ツール` はメイン領域の先頭に控えめに表示されます。サイドバーでは重複表示せず、`新しいレビューを始める` を重要操作として強調しています。
G-4 以降の文書アップロード画面と送信画面では、旧ヒーロー / Co-Pilot を使わず、各ステップのステータスバーと短い見出しで操作します。Step 3 では `送信されるもの / 送信されないもの` を確認してから `レビューを実行` します。
G-3 以降の匿名化結果確認画面では、マスク判断候補がある場合に画面上部へ優先表示されます。候補を判断して `匿名化結果を再生成` するまで、送信準備は完了できません。
G-2 以降のレビュー結果画面では、旧来の AI Display Director や文書別詳細トグルではなく、まず `N 件の指摘` の大きなサマリと `対応すべき指摘` カードを確認します。章再分析、文書構成チェック、将来リスク、使い方は `補助で見るもの` に集約されています。

評価用URL:

```text
https://codexappver2-qicnqzha2kbcadvj63j63e.streamlit.app/
```

## 1. このモックで確認してほしいこと

本ツールは、アップロードした技術文書を匿名化し、業界標準・公的ガイドラインに基づいて
構成・品質・リスクをレビューする支援ツールです。

確認してほしい主な観点は次のとおりです。

- 文書をアップロードしてからレビュー結果を得るまでの操作が分かりやすいか。
- 匿名化結果、送信可否、最終承認の流れが不安なく理解できるか。
- 文書構成チェック、概要レビュー、章別レビュー、深堀レビューの表示が実務に役立つか。
- 複数ファイルを1つの文書として扱う操作感が自然か。
- トークン消費や待ち時間に関する注意表示が十分か。

## 2. 事前に共有する注意事項

評価開始前に、試用者へ次の点を説明してください。

- 最新版を確認する場合は、Streamlit Cloud の Reboot 後、ブラウザをハードリフレッシュしてください。
- まずはリポジトリ内のモック用文書で確認してください。
- 実業務文書を使う場合は、社外秘・個人情報・認証情報を含む文書を不用意に投入しないでください。
- 外部 LLM へ送信されるのは匿名化済みテキストのみですが、送信前に必ず匿名化結果を確認してください。
- 大量の PDF や長大な手順書を投入すると、Gemma/Gemini 側のトークン消費と待ち時間が増えます。
- スキャン PDF や画像中心の資料は、本文抽出・OCR の精度に依存します。最初はテキストが取れる文書で確認してください。
- Word / Excel / PowerPoint は、本文・表・スライド上のテキストを抽出してレビューします。ページ見た目、セル結合、図形配置、矢印、色などの視覚表現は完全には保持されないため、必要に応じて原本と照合してください。
- Excel はシート数、非表示シート、数式、リンク、結合セル、コメントをローカルで診断します。コメント本文も匿名化・機密度判定の対象になります。

## 3. 推奨テスト資料

最初の動作確認では、以下のいずれかを使ってください。

| 用途 | ファイル | 期待する確認内容 |
|---|---|---|
| 標準的な設計書レビュー | `mocks/mock_基本設計書_HikariAuth_v1.docx` | 章別概要、構成不足、深堀候補、レビュー指摘の表示 |
| 新人が書きそうな不十分な文書 | `doc/mock_newcomer_basic_design_notes.txt` | 不足観点、必須要素不足、構成整理提案が出ること |
| 複数 PDF の一括レビュー | 評価者が用意した同一種類の PDF 群 | 複数ファイルを1つの文書として扱えること、送信規模が表示されること |

実業務文書で試す場合も、最初は1ファイルまたは小さめの章単位に分けて確認することを推奨します。

## 4. 画面構成

### 4.1 メイン画面

メイン画面は次の流れで構成されています。

1. ステップ 1 — 文書アップロード
2. ステップ 2 — 匿名化結果プレビュー
3. ステップ 3 — 送信
4. レビュー結果表示

各ステップの上部には、現在の状態を示すステータスバーと短いステップ見出しが表示されます。
ステップ 1 は文書アップロード、ステップ 2 は匿名化結果確認、ステップ 3 は外部送信前の最終確認、レビュー結果は指摘カード確認に集中する構成です。
主な状態は「準備中」「匿名化済み」「確認待ち」「送信準備完了」「レビュー中」「レビュー完了」です。
送信できない文書がある場合は「送信不可」が赤色で表示されます。

レビュー完了後は、画面上部に `N 件の指摘` の大きなサマリが表示されます。続いて `対応すべき指摘` カードを確認し、章再分析・文書構成チェック・将来リスク・使い方は `補助で見るもの` から必要なものだけ開きます。

### 4.2 サイドバー

通常操作で使う項目は少なく抑えています。

| 項目 | 通常の扱い |
|---|---|
| 動作環境 | 現在のレビュー LLM、匿名化方式、機密度判定方式を確認します。 |
| 新しいレビューを始める | アップロード済みの文書、マスク判断、レビュー結果をクリアして最初からやり直します。 |
| 詳細設定 | 通常は開きません。文書種別、マスク辞書、R-M、開発者モードを確認する場合だけ使います。 |
| 採用規格 | レビュー基準の根拠を確認できます。 |

文書種別は詳細設定の中にあります。通常は「自動判定」のままで問題ありません。
自動判定が実際の文書種別と異なる場合だけ手動で変更してください。

動作環境のレビュー LLM は、画面上では `Gemma 4 (Gemini API) / gemma-4-31b-it`
のように、利用者向けの名称と実モデル名を組み合わせて表示されます。

## 5. 基本操作

### 5.1 ステップ 1 — 文書アップロード

1. `レビュー対象ファイル` からレビューしたい文書を選びます。
2. 同一種類の文書であれば、複数ファイルをまとめてアップロードできます。
3. 前回文書との再レビュー比較を行う場合だけ、`前回のレビュー結果と比較する` を開き、前回保存した `remediation_plan_YYYYMMDD_HHMM.json` を選びます。旧 `remediation_plan.json` も読み込めます。
4. 「匿名化してプレビュー」を押します。

補足:

- 設計書と手順書など、異なる種類の文書を一度に混在させる使い方は推奨しません。
- 複数 PDF は「1つのレビュー対象を複数ファイルで構成している」として扱います。
- PowerPoint はスライド本文とノート、Excel はシートごとの行テキスト、Word は本文 XML を中心に抽出します。図表やレイアウトの意味は補助情報として扱ってください。
- Excel ファイルでは、各文書カード内に「Excelブック診断（ローカル抽出）」が表示されます。非表示シートやコメントがある場合は、外部レビュー送信前に内容を確認してください。
- 選択中ファイルは画面上に一覧表示されます。同一内容の重複ファイルは一覧上で警告され、処理が止まる場合があります。

### 5.2 ステップ 2 — 匿名化結果プレビュー

匿名化が完了すると、以下の情報が表示されます。

| 表示 | 見るポイント |
|---|---|
| サマリチップ | 安全、要確認、候補数、本文トークン数を確認します。 |
| マスク判断候補 | 未確定の固有名詞がある場合、画面上部に表示されます。外部 LLM に送る前に「マスクする / マスクしない」を選びます。 |
| 匿名化結果を再生成 | マスク候補の判断を反映して、匿名化済みテキストを再作成します。 |
| 送信準備を完了する | マスク判断候補が残っていない場合に有効になります。押すと Step 3 の最終承認へ進みます。 |
| 文書ごとの詳細 | 必要な文書だけ開き、判定理由、匿名化後テキスト、置換一覧を確認します。 |
| 前回計画との照合 | 「前回文書の再レビューを使う」をオンにして前回の修正計画JSONを読み込んだ場合だけ、今回文書に改善要素が反映されているかを確認します。 |

マスク判断候補がある間は「送信準備を完了する」は無効です。
候補ごとに判断したあと、「匿名化結果を再生成」を押すと判断が反映され、送信準備へ進めます。
再生成が必要な候補がない場合、「匿名化結果を再生成」は無効化されます。これは正常な状態です。

Gemma4送信前トークン予算では、予定call数、本文トークン、入力合計に加え、文書別の送信規模、Free tier対策の最低待機時間、推奨レビュー分割案を確認できます。
分割推奨が出た場合は、ファイル数を減らす、章単位に分ける、別紙ログや不要なOCR行を外す、といった調整を検討してください。

### 5.3 マスク候補が出た場合

未確定の固有名詞や企業名候補が検出されると、Step 2 の上部に「マスクするか判断が必要な語」として候補が表示されます。

1. 文脈を確認します。
2. 外部 LLM に送るべきでない語は「マスクする」を選びます。
3. 問題ない語は「マスクしない」を選びます。
4. 「匿名化結果を再生成」を押します。
5. 再生成後、必要に応じて文書ごとの詳細から匿名化後テキストを再確認します。
6. 「送信準備を完了する」を押し、Step 3 の最終承認へ進みます。

迷う場合は、情報漏えい防止を優先して「マスクする」を選んでください。

### 5.4 ステップ 3 — 送信

1. 送信先として表示される LLM プロバイダ名、予定 call 数、本文トークン数を確認します。
2. `送信されるもの` と `送信されないもの` の2カラムを確認します。外部へ送られるのは匿名化済みテキストとレビュー基準プロンプトで、原文ファイルやファイル binary は送られません。
3. 内容に問題がなければ `上記の内容で送信することを承認します。` にチェックを入れます。
4. `レビューを実行` を押します。戻って確認したい場合は `ステップ 2 に戻る` を押します。

送信ボタンが押せない場合は、次のいずれかです。

- 承認チェックが入っていない。
- 送信禁止の文書が含まれている。
- アップロードまたは匿名化プレビューがまだ完了していない。

### 5.5 レビュー実行中

レビュー送信後、進捗バーとステータスが表示されます。
複数ファイルや長文の場合は、分割 call になるため時間がかかることがあります。

途中でエラーが出た場合は、画面のエラーメッセージをそのまま記録してください。
Gemma/Gemini 側の一時的な HTTP 500 やクォータ制限は、時間を置いて再試行すると解消する場合があります。

## 6. レビュー結果の見方

### 6.1 文書構成チェック

文書全体として不足している観点を確認します。

| 分類 | 意味 |
|---|---|
| 不足観点 | その観点に関係する記述自体が見当たらないものです。 |
| 必須要素不足 | 観点の章や記述はあるが、目的・対象外・責任分担などの中身が足りないものです。 |
| 構成整理の提案 | 記述はありそうだが、見出し・章・粒度を整理した方がよいものです。 |

ここで表示される内容は、章番号そのものではなく、設計書として確認したい観点に基づく不足です。

### 6.2 文書全体の概要

LLM が文書全体の目的、対象、懸念点を要約します。
この要約は、後続の章別レビューを読む前の前提確認として使います。

文書内に「目的」セクションがある場合は、その記載箇所も表示されます。
文書に目的が見当たらない場合や、本文から推定される目的と記載内容にずれがある場合は、
目的追記または目的と本文の整合確認を促す表示が出ます。

### 6.3 修正アクションプラン

レビュー指摘と文書構成チェックをもとに、修正方針、追記テンプレート、再レビュー条件が自動で整理されます。
この処理は既存レビュー結果からローカルで生成し、外部 LLM への追加送信は行いません。

修正アクションプランでは、まず高重要度のカードを担当者へ割り当て、次に追記テンプレートを文書へ反映します。
その後、カードごとの再レビュー条件に沿って、対象章または対象観点だけを確認してください。

「📒 修正計画 JSON」は、`対応すべき指摘` セクション右側に表示されます。
この JSON は人に渡す作業依頼書や監査ログではなく、次回の再レビュー時に修正文書と一緒に読み込ませるための台帳です。
次回比較を行う予定がある場合だけ保存してください。次回読み込むと、前回指摘の対象章・キーワード・追記候補が今回文書に反映されているかをローカルで照合します。

元のレビュー指摘や文書別詳細は、通常画面では主導線から外しています。
通常は `対応すべき指摘` カードを確認すれば足ります。開発者モードでは、監査・照合用の証跡ZIPやプロンプト情報を確認できます。

### 6.4 先読みレビュー

レビュー結果の補助として、`補助で見るもの` の `将来の障害リスク — 主要指摘の先にある予防策を確認` を開けます。
この処理はローカル解析であり、外部 LLM への追加送信は行いません。

先読みレビューでは、以下を確認できます。

| 表示 | 見るポイント |
|---|---|
| 曖昧表現の未確定チェック | 「必要に応じて」「原則」「適宜」などの表現に、判断条件・責任者・期限が不足していないか確認します。 |
| 読み手別の誤読リスクマップ | 初任SE、二次運用者、監査人、上長・承認者の観点で、読みづらい箇所や不足しやすい観点を確認します。 |
| 未来障害プレモーテム | DR、監視、認証、通信経路などについて、将来障害につながりやすい道筋と次の一手を確認します。修正計画と重複する根拠欄は省略し、将来リスクに絞って表示します。 |
| メタレビュー | 開発者モード時のみ表示します。各指摘が「有効」「言い過ぎ」「不要」「見落としあり」のどれに近いか、上司・管理者テスト用に記録できます。 |

「この指摘の精度を評価」は、開発者モード時のみ表示されるセッション内の品質フィードバックです。
通常の文書レビュー結果には影響しません。

開発者モード ON の場合だけ、レビュー結果側に「証跡エクスポート — 監査ログを共有・保存するときに開く」が表示されます。
匿名化済みテキスト、マスク候補、送信対象ログ、レビュー結果を 1 つの `audit_log_YYYYMMDD_HHMM.zip` として保存できます。
通常操作では表示されません。検証ログを共有する場合だけ利用してください。

### 6.5 章単位の追加レビュー

`補助で見るもの` の `章単位の追加レビュー` を開くと、アップロード文書から検出した章ごとに概要と再分析ボタンが表示されます。

「深堀候補」と表示された章は、追加確認の優先度が高い章です。
指摘カードに対象章を紐づけられる場合は、カード内にも `章を再分析` ボタンが表示されます。
通常モードでは、最初の深堀候補章だけ再分析ボタンが有効になります。
これはトークン消費を抑えるための仕様です。

### 6.6 深堀レビュー

指摘カードの `章を再分析`、または `補助で見るもの` の `章単位の追加レビュー` から「この章を再分析」を押すと、その章について追加レビューが実行されます。
深堀で追加された指摘は `対応すべき指摘` カードへ合流し、深堀由来カードにはバッジが表示されます。

深堀レビューは、概要レビューと矛盾しないように、概要レビューで懸念がある章を優先して使います。
深堀を複数回行う場合、2回目以降は「追加確認結果」として扱い、同じサマリを繰り返さない方針です。

### 6.7 マスク判断履歴

ページ下部の「全期間のマスク判断履歴と推奨」は、過去にマスクした語や素通しした語の傾向を確認するための運用補助です。
履歴が多い場合でも画面が長くなりすぎないよう、初期表示では判断回数の多い上位10件だけを表示します。

11件以上の履歴がある場合は「もっと見る」で全件を展開できます。
全件表示後は「上位10件のみ表示に戻す」で初期表示に戻せます。

### 6.8 指摘ID

レビュー指摘には、文書種別に応じた接頭辞が付きます。

| 接頭辞 | 意味 |
|---|---|
| D | Design。設計書レビューの指摘です。 |
| CR | Change Runbook。変更・切替手順書レビューの指摘です。 |
| OR | Operations Runbook。保守・運用手順書レビューの指摘です。 |
| NC | Network Config。ネットワーク機器 Config レビューの指摘です。 |
| SC | Source Code。ソースコードレビューの指摘です。 |

番号はこのレビュー内での管理番号です。社内正式チケット番号ではありません。

## 7. 推奨テストシナリオ

### 7.1 標準シナリオ

1. `mocks/mock_基本設計書_HikariAuth_v1.docx` をアップロードします。
2. 文書種別は通常変更不要です。誤判定が疑われる場合だけ、サイドバーの詳細設定を開いて変更します。
3. 「匿名化してプレビュー」を押します。
4. Step 2 のサマリ、マスク判断候補、文書別の詳細を確認します。
5. `送信準備を完了する` を押して Step 3 に進みます。
6. `送信されるもの / 送信されないもの` を確認し、承認チェックを入れて `レビューを実行` を押します。
7. レビュー結果では、まず `N 件の指摘` サマリと `対応すべき指摘` カードを確認します。追加確認が必要な場合は、`補助で見るもの` から章単位の追加レビュー、障害シナリオ、文書構成チェック、使い方を開きます。

確認ポイント:

- `対応すべき指摘` カードが重要度順に表示され、最初の高重要度カードが展開されているか。
- 指摘カードに章を紐づけられる場合、カード内に `章を再分析` ボタンが表示されるか。
- `補助で見るもの` に、文書構成チェック、章単位の追加レビュー、将来の障害リスク、修正計画の使い方がまとまっているか。
- 匿名化検知内容に `ローカル機密度ゲート` と日本語で表示されるか。
- 文書全体の概要に、目的・内容要約・全体評価が表示されるか。
- 修正アクションプランに、追記テンプレートと再レビュー条件が表示されるか。
- 障害シナリオと予防策に、曖昧表現・読み手別リスク・未来障害予兆が分かれて表示されるか。
- 「📒 再レビュー用の修正計画JSONを保存」が、次回比較用として分かりやすい位置に表示されるか。
- 指摘内容が実務レビューとして納得できる粒度か。

### 7.2 新人作成風ドキュメントのシナリオ

1. `doc/mock_newcomer_basic_design_notes.txt` をアップロードします。
2. 「匿名化してプレビュー」を押します。
3. Step 2 で `送信準備を完了する` を押し、Step 3 で承認後に `レビューを実行` を押します。
4. 文書構成チェックを確認します。

確認ポイント:

- 目的、対象外、責任分担、承認、DR、監視などの不足が分かりやすく表示されるか。
- 「不足観点」「必須要素不足」「構成整理の提案」の違いが理解しやすいか。
- 新人向けの指摘として厳しすぎないか、または不足が明確に伝わるか。

### 7.3 複数ファイル一括レビューのシナリオ

1. 同一種類の設計書 PDF を複数ファイル選びます。
2. まとめてアップロードします。
3. 「匿名化してプレビュー」を押します。
4. 送信規模、予定 call 数、分割推奨の有無を確認します。
5. 必要に応じて、ファイル数を減らして再試行します。

確認ポイント:

- ファイル名順、章番号順に見やすく表示されるか。
- 合算トークン規模が判断しやすいか。
- 文書別の送信規模と推奨レビュー分割案が判断しやすいか。
- 大量文書投入時に「分割推奨」が納得できる形で表示されるか。

### 7.4 再レビュー比較のシナリオ

1. 初回レビューで再レビュー比較を使う予定がある場合は、レビュー結果の「📒 再レビュー用の修正計画JSONを保存」からJSONを保存します。
2. 文書を修正した想定で、修正版の文書をアップロードします。
3. 「前回文書の再レビューを使う」をオンにし、「前回の修正計画JSONを読み込む」から、1で保存したJSONを選びます。
4. 「匿名化してプレビュー」を押します。
5. 「前回の修正計画JSONと今回文書を照合しました」の内容を確認します。
6. 必要に応じて Step 3 で承認後に `レビューを実行` し、再レビュー用の修正計画JSONを次回比較用に保存します。

確認ポイント:

- 改善あり、一部改善、未確認、要確認の表示が分かりやすいか。
- JSONが作業依頼書ではなく、再レビュー比較の台帳として理解できるか。
- 前回指摘が解消したかどうかを、今回レビュー結果と合わせて判断できるか。

## 8. トラブルシュート

| 症状 | 主な原因 | 対処 |
|---|---|---|
| アプリがスリープしている | Streamlit Community Cloud のスリープ | 画面の起動ボタンを押して復帰を待つ。 |
| `レビューを実行` が押せない | 承認未チェック、送信禁止、要確認残り | ステップ 2 とステップ 3 の表示を確認する。 |
| 匿名化結果を再生成できない | 再生成対象の候補がない | 正常。安全判定ならそのまま送信確認へ進む。 |
| PDF の本文が少ない | スキャン PDF または画像中心の資料 | OCR 済み PDF、テキスト化済み資料、または章単位のテキストで再確認する。 |
| レビューが遅い | 長文・複数ファイル・分割 call | 小さい単位に分ける。まずモック文書で確認する。 |
| 画面表示が古い | 自動再デプロイ前、またはブラウザキャッシュ | Streamlit Cloud で Reboot し、ブラウザをハードリフレッシュする。 |
| Gemini/Gemma の HTTP 500 | 外部 LLM 側の一時エラー、プロンプト/出力形式の不整合 | 画面メッセージを記録し、少し待って再実行する。再現する場合はログ JSON を共有する。 |
| クォータ制限らしきエラー | 無料枠の制限 | 時間を置く。大容量資料を避ける。 |

## 9. 評価者に依頼したいフィードバック

評価後、次の観点でコメントをもらうと改善に直結します。

記録には `docs/admin_test_feedback_template.md` を使うと、再現手順、重要度、次アクションを整理しやすくなります。

- 操作手順で迷った箇所はどこか。
- 匿名化・最終承認の説明に不安はないか。
- レビュー指摘は実務上使える深さか。
- 指摘が厳しすぎる、または甘すぎる箇所はあるか。
- 文書構成チェックと章別レビューの内容に矛盾を感じる箇所はあるか。
- 複数ファイルを1文書として扱う表示は分かりやすいか。
- トークン消費・分割推奨の表示は意思決定に役立つか。
- 追加してほしいレビュー観点はあるか。

## 10. 終了時の操作

確認が終わったら、必要に応じて以下を行います。

1. 監査ログを共有・保存する場合だけ、開発者モードを ON にし、レビュー結果側の「証跡エクスポート — 監査ログを共有・保存するときに開く」から `audit_log_YYYYMMDD_HHMM.zip` を保存します。
2. サイドバー上部の「新しいレビューを始める」を押します。
3. ブラウザを閉じます。

ログ JSON には、匿名化後に LLM へ送る文章やプロンプトが含まれます。
共有する場合は、内容を確認してから扱ってください。
