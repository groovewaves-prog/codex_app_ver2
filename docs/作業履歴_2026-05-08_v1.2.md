# 第 3 章 実装した機能の詳細

## 3.1 R-X: アップローダリセット + 重複検出

### 課題

- リセットボタンを押しても、視覚的にアップロード済みファイルが残っているように見える
- 同じファイルを複数アップロードしても検出できない

### 解決方法

- **R-X-1 (リセット視覚化)**: `uploader_key` を `uuid.uuid4().hex[:8]` で動的生成し、リセット時に新しいキーを生成。Streamlit の file_uploader は key が変わると内部状態がクリアされる。
- **R-X-2 (重複検出)**: SHA256 ハッシュで全ファイルを検証し、重複があれば `st.error` で中断。

### 主要決定事項

- Q1=B (動的 uploader_key)
- Q2=B (SHA256 ハッシュ判定)
- Q3=A (重複時は完全拒否、ユーザに修正を促す)

### 影響ファイル

- `streamlit_app.py`: `_get_uploads()`, `_detect_duplicate_uploads()` 追加

## 3.2 R-Y: 深堀レビュー (文書ごと)

### 課題

LLM レビューの初回応答に納得できない場合、特定文書のみ深く再レビューしたい。

### 解決方法

- 各文書のヘッダ近くに「🔬 この文書を深堀」ボタンを配置
- 深堀時は、PDF + 既存の指摘 + 「もっと深く分析せよ」というプロンプトを LLM に送信
- 結果は同じ文書グループ内に「📌 深堀結果」expander として追加表示 (累積可能、#1, #2, ...)

### 主要決定事項

- Q4=A (文書全体を深堀対象)
- Q5=A (各文書ヘッダ近くにボタン、Step 4 内に統合)
  - 一度 Q5=B (別 Step として selectbox) で実装したが、ユーザの意図を再確認して Q5=A に修正
- Q6=A (深堀結果は既存の文書グループに追加)
- Q7=A (PDF + 既存指摘 + 深堀指示を全部送る)

### 影響ファイル

- `secure_review/reviewer.py`: `ReviewProvider.review()` に `deep_dive_target` と `existing_issues` 引数を追加
- `streamlit_app.py`: 文書ごとグループ化と深堀ボタン
- `streamlit_audit_ui.py`: log JSON に deep_dive_results を含める

## 3.5 Q12-Q13: 文書チェックボタン新設

### 課題

「レビューに送信」を押すと、apply_user_decisions + render_session_summary + render_log_export_button + provider.review() が一気に走る。LLM 送信中なのに永続化ボタンや DL ボタンが画面に表示されている違和感があった。

### 解決方法

2 段階フローに分離:

```
[匿名化してプレビュー] → uncertain candidates 判断 →
[📋 文書チェック]        ← ローカル処理 (apply_user_decisions + サマリ + DL 表示)
[レビューに送信]          ← LLM 送信のみ (文書チェック後のみ有効)
```

- 「📋 文書チェック」: secondary ボタン
- 「レビューに送信」: primary ボタン、`docs_checked == False` なら disabled
- ステータスメッセージで利用者に次のアクションを案内
- ローカルで匿名化を確認してから外部送信、というツール本来の思想を UX 上も表現

### 主要決定事項

- Q12=A (「📋 文書チェック」が左、「レビューに送信」が右)
- Q13=A (左 → 右に進む配置)

### 影響ファイル

- `streamlit_app.py`: ボタン構造の刷新、send_clicked のロジック分離、`docs_checked` フラグの導入

## 3.6 Q14-Q15: 開発者モードで実験 UI 分離

### 課題

実務機能 (レビュー結果、深堀、判断履歴、永続化) と試験機能 (NER Diagnostics、gBizINFO Diagnostics、プロンプトプレビュー、生レスポンス) が同じ画面に混在しており、実務ユーザの認知負荷が高い。

### 解決方法

サイドバーに「⚙️ 開発者モード」トグルを追加 (デフォルト OFF)。

| 開発者モード OFF (デフォルト) | 開発者モード ON |
|---|---|
| 実務機能のみ表示 | 実務機能 + 実験/デバッグ UI を見出し付きで表示 |

開発者モード OFF 時に非表示になる UI:
- プロンプトプレビュー (先頭 2000 文字)
- LLM の生レスポンス (デバッグ用)
- 日本語 NER Diagnostics (R-M 実験)
- gBizINFO 検索 Diagnostics (R-M Phase 2 実験)

「全期間のマスク判断履歴と推奨」は **運用機能** として常時表示 (実務扱い)。

環境変数 `DEVELOPER_MODE_DEFAULT=true` で初期値を ON にできる (開発環境用)。

### 主要決定事項

- Q14=A (デフォルト OFF、環境変数で初期値変更可能)
- Q15=A (全期間履歴は実務扱い、常時表示)

### 影響ファイル

- `streamlit_app.py`: 開発者モードトグル、4 つの実験 expander を `if developer_mode` で囲む

# 第 5 章 重要な議論・決定事項

## 5.1 ユーザによる本質的な軌道修正

セッション中、ユーザは **AI エージェントだけでは到達しづらい本質的な指摘** を複数回行った。これらが本セッションの方向性を大きく決定した。

### 5.1.1 「評価ツールを作る前に物差しを確立する」

**経緯**: Claude が「LLM 評価が薄い → プロンプト改善 → UI 改修」という線形の対症療法に向かいかけた時、ユーザは:

> 「設計書なり、手順書なり、『文書のこうあるべき』構造というのがあると思います。この構造を決定しない限り、評価は曖昧になると思います。まず、文書構造を他者の知見 (ITIL など) に基づき決定しませんか。」

と提案。これにより方針が大きく変わり、構造定義書 v0.1/v0.2 の作成に至った。

**含意**: **根拠のない物差しでは、根拠のある評価はできない**。業界標準に依拠することで、評価の正当性を主張できる。

### 5.1.2 「実務機能と試験機能の混在」

**経緯**: ステップ 4 のレビュー結果画面に、実務機能 (指摘表示、深堀ボタン、判断履歴) と試験機能 (NER Diagnostics、gBizINFO Diagnostics、プロンプトプレビュー、生レスポンス) が混在していることをユーザが指摘:

> 「実務に使う機能と試験機能が混然としています。今後も試験 (画面では実験) を要するのなら、UI の表示方法を分けるなどしたい。」

**含意**: 実務ユーザの 99% は実験機能を使わない。デフォルト OFF で完全に隠す方が UX として優れる。

### 5.1.3 「文書チェックボタンによる UX 改善」

**経緯**:

> 「『レビュー送信』ボタンを押さないと『マスク確定』『watchlist』『素通し確定』『結果ログをダウンロード』の各ボタンがクリックできません。これは LLM へプロンプト送信中の行為となるので、違和感があります。」

**含意**: ローカル処理 (匿名化確認) と外部通信 (LLM 送信) を UI 上も明確に分離することで、ツール本来の設計思想を表現できる。

## 5.2 「AI エージェントは人間の思考をアシストする」原則

ユーザは複数回:

> 「AI エージェントは人間の思考をアシストするという本質を忘れず、実装をお願いします。」

と強調。これは以下のスタイルで実践された:

1. **実装に飛び込む前に、方針を提示**: Q1〜Q15 のような選択肢形式で論点を整理
2. **ユーザが判断・指示、AI が実装**: 推奨案を提示しつつ、最終判断はユーザに委ねる
3. **ユーザの指摘を本質的に受け止める**: 表層の対応ではなく、根本原因に立ち戻る

```python
@dataclass(frozen=True)

@dataclass(frozen=True)
class StandardChapter:
    chapter_id: str            # "ch1"
    chapter_name: str          # "はじめに"
    purpose: str
    items: tuple[ChapterChecklistItem, ...]


DESIGN_DOC_STRUCTURE_V0_2: tuple[StandardChapter, ...] = (
    StandardChapter(chapter_id="ch1", ...),
    # 全 15 章
)
```

## 9.1 課題 1: マスク判断サマリ expander が永続化ボタン押下で折りたたまれる

### 症状

「📊 本セッションのマスク判断サマリ」expander を展開した状態で、内部の永続化ボタン (📥 マスク確定 / 👁️ 都度判断 / 🟢 素通し確定) のいずれかを押すと、expander が **強制的に折りたたまれる**。次の語の判断作業に連続して進めない。

### 原因

Streamlit の `st.expander` は、ボタン押下による rerun のたびに `expanded` パラメータの初期値に戻る。現状 `expanded=False` で実装されているため、ユーザが手動で開いた状態が rerun で消失する。

### 対応方針

- **方針 A**: `session_state` に展開状態を保持し、`expanded=st.session_state.summary_expanded` で制御
- **方針 B**: `st.expander` をやめて `st.container` + ヘッダで常時表示にする (展開・折りたたみ機能を捨てる)
- **方針 C**: 「📥 マスク判断サマリは閉じない」設定を session_state に追加し、ボタン押下時に展開状態を復元

推奨は **方針 A**。expander の利便性 (折りたたみ可能) を維持しつつ、勝手に閉じないようにする。

### 影響範囲

- `streamlit_audit_ui.py`: `render_session_summary()` 内の `with st.expander(...)`

### 想定作業時間

## 9.2 課題 2: Gemini API 呼び出し失敗 (根本解決を要する)

### 症状

「レビューに送信」ボタン押下後、以下のエラーが発生:

```
Gemini (gemini-2.5-flash) could not be reached. See server logs for details.
```

### 結果ログ JSON の分析 (重要)

ユーザが提供した結果ログ JSON を分析した結果、**入力サイズは問題ではない** ことが判明した。

| 項目 | 値 | Gemini 2.5 Flash の上限 |
|---|---|---|
| 12 文書の本文合計 | 13,136 chars (≈ 6,568 tokens) | 入力上限: 1,000,000 tokens |
| 入力サイズ占有率 | **0.66%** | — |

### 9.2.1 推測されていた原因 (Streamlit ログ提供前)

| 候補 | 可能性 | 根拠 |
|---|---|---|
| A. Gemini の出力 token 上限 (8,192) に到達 | 🔥 最有力と推測した | 12 文書 × 複数指摘 + summary を JSON で返すと出力が長くなる |
| B. Free tier のレート制限 (10 RPM) | 中 | 連続テスト時に発生する可能性 |
| C. ネットワーク一時障害 | 中 | Gemini 側の一時的な問題 |
| D. responseSchema が複雑で内部エラー | 中 | structured output の特殊な失敗 |

### 9.2.2 実際のエラー (Streamlit ログから判明)

ユーザが Streamlit Cloud のログを共有してくれた結果、**2 種類のエラーが混在** していたことが判明した。当初の最有力推測 (出力 token 上限) は **的外れ** だった。

#### エラー A: Gemini HTTP 503 (Gemini サーバー側の過負荷)

```
Gemini (gemini-2.5-flash) HTTP 503
"code": 503
"message": "This model is currently experiencing high demand. 
            Spikes in demand are usually temporary. Please try again later."
"status": "UNAVAILABLE"
```

- 発生時刻: 15:04 (古いコード版の動作中)
- 3 回連続で発生
- **これは Google 側のサーバー過負荷**。free tier 固有ではなく、有料プランでも発生する。
- クライアントの対応手段は **リトライのみ** (指数バックオフでサーバの負荷分散に協力)。

#### エラー B: HTTP read timeout (クライアント側の受信タイムアウト)

```
Gemini (gemini-2.5-flash) transport error: The read operation timed out
```

- 発生時刻: 17:27 以降 (最新コード `7e40455` + 今日の修正反映後の動作中)
- 4 回連続で発生
- **これはクライアント側の HTTP read timeout**。urllib の timeout 設定を Gemini の応答生成時間が超過。
- 12 文書を 1 API call で処理する現設計では、応答生成に時間がかかりすぎる。

#### 真因と対策の対応関係 (再整理)

| 戦略 | 対 503 (エラー A) | 対 read timeout (エラー B) |
|---|---|---|
| A. chunking (1 文書 1 API call) | ✅ 各 call が小さく、503 になる確率低下 | ✅ 各 call の応答時間が 5-10 秒 → timeout の心配なし |
| B. max_output_tokens 増加 | △ 直接効果なし | ✅ 応答打ち切り防止 |
| C. リトライ機構 + 指数バックオフ | ✅✅ **これしか対策なし** | ✅ 一時的なネットワーク問題を吸収 |
| D. timeout 値見直し | △ | ✅ 余裕確保 |

→ **A + B + C + D の組み合わせで両方を根本解決** できる見通し。

#### lesson learned

**ログ提供前**: 「出力 token 超過」を最有力と推測 (誤り)。
**ログ提供後**: 「サーバー側 503」+「クライアント側 timeout」の 2 種類が真因と判明。

これは「人間の思考が AI の作業の方向を正した」典型例である。AI エージェントが推測だけで実装に進んでいたら、的外れな対策をしていた可能性が高い。**実データ (ログ) を確認することの重要性** を改めて確認した。

### 根本解決の方向性

#### 戦略 A: 文書ごとに API call を分割 (chunking) — 推奨

```
旧フロー: 12 文書を 1 API call にまとめる
   ↓ 出力 token に余裕なし、失敗時に全滅、原因切り分け困難

新フロー: 1 文書 = 1 API call (12 回呼び出し)
   ↓ 各応答は小さく安定、失敗時は文書単位でリトライ
   ↓ 進捗を spinner で表示可能 (例: "3/12 文書を処理中...")
   ↓
全体集約: 1 API call
   - 各文書の結果を input にして missing_chapters (欠落章) を判定
```

この設計は **Phase 4〜6 の `checklist_results` 設計と完全に親和** する。各文書ごとに 15 章のチェック項目を評価する設計なら、文書ごとの API call が自然な単位となる。

利点:
- 各 API call の入出力サイズが小さく安定
- 失敗時の影響が文書単位に限定 (リトライ可能)
- 進捗の可視化が可能
- Free tier のレート制限内で確実に処理

デメリット:
- API call 回数が増える (12 + 1 = 13 回)
- 全体処理時間が長くなる (60〜120 秒程度)
- Free tier の RPM 制限 (10 RPM) に引っかかる可能性 → リトライ間隔の制御が必要

#### 戦略 B: 出力 token 上限を明示的に増やす

`max_output_tokens=32768` (Flash の最大) に設定。`reviewer.py` の Gemini プロバイダ呼び出しで明示的に指定。

#### 戦略 C: リトライ機構 + 詳細エラーログ

- 失敗時の自動リトライ (指数バックオフ、最大 3 回)
- エラー種別の切り分け:
  - HTTP 4xx / 5xx
  - timeout
  - JSON parse error
  - レート制限 (429)
  - 出力 token 不足
- ユーザに分かりやすいエラーメッセージ + 推奨対応 (「しばらく待って再実行」「文書数を減らす」等)

### 推奨実装方針 (明日)

**A + B + C の組み合わせ**:

1. **A** (chunking): 文書ごとの API call に変更。これは Phase 5 (LLM スキーマ拡張) と統合して実装
2. **B** (出力 token 増加): 念のため `max_output_tokens` を最大値に設定
3. **C** (リトライ + エラーログ): 安定性向上

これにより、Gemini エラーは根本解決される (各 call が小さく、失敗時はリトライ可能)。

### 影響範囲

- `secure_review/reviewer.py`:
  - Gemini プロバイダの `review()` メソッドを文書ごとループに変更
  - `max_output_tokens` を明示的に指定
  - リトライ機構 + エラー切り分け
- `streamlit_app.py`:
  - 進捗バー (例: "3/12 文書を処理中...") の表示
  - エラー時の詳細メッセージ表示
